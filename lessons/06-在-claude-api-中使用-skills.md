# 06 在 Claude API 中使用 Skills

## 这章讲什么

前面几章，你看到的是 skill 的概念和结构。这一章开始进入工程实践：

> 如果不用图形界面，而是直接通过 API 使用 skills，会发生什么？

原课的核心结论很明确：

- `Skill` 的格式本身不变
- 但你所在的运行环境会变
- 一旦环境变了，你就得自己补齐 Claude.ai 帮你封装好的那些东西

## 这一章的阅读重点

这章不是让你死记某个 notebook 怎么跑。

它真正要你建立的是一个工程意识：

- skill 可以迁移
- 但运行环境不会自动补齐

## 这里第一次出现的术语怎么理解

- `API`：程序和程序之间沟通的接口
- `文件系统`：程序读写文件的环境
- `容器`：一种隔离运行环境，可以理解成给程序准备的一台“小电脑”

## 为什么 API 场景更难

在 `Claude.ai` 或 `Claude Desktop` 里，很多能力是默认给好的，比如：

- 执行代码
- 创建文件
- 使用某种文件系统

但当你直接用 `Claude API` 时，这些东西不会凭空出现。

也就是说，在 API 场景下，你不仅是在“调用模型”，你还在搭一套让 skill 能发挥作用的执行环境。

这也是为什么很多人在 Web 端觉得“挺好用”，一到 API 端就突然觉得麻烦了很多。不是 skill 变了，而是你开始要自己补基础设施了。

## 原课强调的两个关键点

### 1. Claude.ai 里的 skill，不会自动同步到 API

很多人第一次接触这里会误会：

“我在 Claude 里已经做好的 skill，API 里应该也能直接用吧？”

原课明确告诉你：不是这样的。

不同环境里的 skill，需要你自己管理和接入。

### 2. API 需要你自己提供执行能力

如果 skill 需要：

- 读写文件
- 运行脚本
- 生成文档或报表

那你就需要在 API 侧自己提供这些能力，比如：

- Files API
- Code Execution Tool
- 容器或文件系统环境

原课这里讲得非常具体：在 Claude Messages API 里，要让 skills 真正跑起来，至少要把 Code Execution Tool 和 Files API 一起接上。

## 中文读者最容易卡住的点

这一章最容易产生错觉的地方是：

“skill 不是都已经写好了吗，为什么 API 里还这么麻烦？”

原因很简单：

- skill 只是方法和资源组织
- API 环境不天然拥有执行条件

你可以把它理解成：

> 你手里有一本详细的菜谱，但厨房、炉灶、刀具、冰箱都得你自己准备。

而且原课还专门提醒了一个很容易忽略的差别：

- 在 Messages API 的代码执行容器里，有资源限制，也默认没有互联网连接
- 但在 Claude.ai / Claude Desktop 里，代码执行环境的很多能力是平台已经帮你准备好的

这不是“哪个更高级”的问题，而是两边的运行条件本来就不同。

## 更贴近中文开发者的场景例子

比如你想做一个“自动整理客服聊天记录并生成日报”的 skill：

- skill 负责定义怎么分类、怎么总结、怎么输出
- 但如果你走 API 路线，还得自己准备：
  - 聊天记录从哪进来
  - 文件怎么保存
  - 脚本在哪跑
  - 结果怎么导出

这就是 skill 和运行环境必须分开的原因。

前面几章主要在讲：

这一章第一次把你拉到“系统要真正跑起来”这个层面上。它让你看到：方法对了，还不够；环境也必须对。

## 所以这一章真正要你建立的是什么

不是“怎么把 notebook 例子照着抄一遍”。

而是理解这个工程事实：

- skill 可以跨环境迁移
- 但迁移的是“结构和方法”
- 不是自动打包好的运行能力

## 这对你有什么实际启发

如果你未来打算做自己的 agent 应用，应该尽早区分这三层：

- 模型层：模型负责推理和生成
- 能力层：工具、文件系统、执行环境负责做事
- 方法层：skill 负责规定流程和标准

很多系统一开始做不稳，就是把这三层混成一层了。

## 给开发者的补充提醒

原课还提到一些现实问题，比如：

- 成本
- 流式输出
- notebook 演示和生产环境之间的差距
- 请求里需要带上的相关 beta headers

所以如果你不是在做一次性 demo，而是真打算上生产：

- 你要评估调用成本
- 你要考虑延迟
- 你要考虑文件与执行环境的生命周期

## 常见误区

- 误区 1：以为 API 只是“把网页按钮换成代码调用”
- 误区 2：以为 skill 能自动解决运行环境问题
- 误区 3：把 notebook 里的演示流程直接等同于生产环境方案

# 05 如何创建自定义 Skills

## 这章讲什么

到了这一章，课程终于开始回答一个最实际的问题：

> 如果我想自己做一个 skill，应该从哪里开始？

原课用了两个案例：

- 基于讲义生成练习题
- 分析时间序列数据的特征

这两个例子分别代表了两种很典型的 skill 场景：

- 内容生成
- 数据分析

## 读这一章时，先别急着记字段

很多人会把这一章读成“格式规范说明”。

那样读会很干，也容易漏掉重点。

更好的读法是：

> 把它当成一章“怎样把经验写成流程”的课。

## 这里第一次出现的术语怎么理解

- `frontmatter`：写在文档最前面的一小段结构化元信息
- `name`：skill 的名字
- `description`：skill 的功能和使用场景说明

## 一个 skill 至少需要什么

原课反复强调，一个 skill 的最小核心是：

- 一个 `SKILL.md`
- 里面包含 frontmatter
- 至少写清楚 `name` 和 `description`

然后才是主体内容、引用文件、脚本、资源等。

## 为什么 name 和 description 这么重要

很多新手会把重点放在正文步骤写得多不多，但原课特别强调：

- `name` 不只是名字
- `description` 不只是简介

它们还承担一个关键职责：

> 帮助 agent 判断“什么时候应该用这个 skill”。

所以一个好描述，通常不只写“它是什么”，还会写：

- 它什么时候该被用上
- 用户通常会怎么提需求
- 哪些关键词能帮助正确触发

原课还补了几个很具体、很工程化的建议：

- `name` 最好使用小写字母、数字和连字符
- 命名习惯尽量偏动作导向，也就是更像“正在执行某类动作”的风格
- `description` 不只写功能，还要写“什么时候该用它”

## 中文场景例子

假设你想做一个“把直播录音整理成复盘提纲”的 skill：

- 如果 `description` 只写“整理内容”，太泛了
- 如果写成“把直播或课程逐字稿整理成结构化复盘提纲，适用于复盘、备课、知识卡片初稿”，agent 更容易知道什么时候该用它

## 写 skill 时最重要的几条原则

### 1. 先写清楚步骤

如果你想让工作流稳定，就不要只写“帮我做好这件事”，而要写：

- 先做什么
- 再做什么
- 什么时候跳过某一步
- 遇到边界情况怎么办

这和写 SOP 很像。

### 2. 把边界情况说清楚

一个 skill 之所以不稳定，常常不是因为正常情况没写清楚，而是因为：

- 输入缺失怎么办
- 文件格式不符合预期怎么办
- 某一步不可执行时怎么办

原课建议把这些条件写明白。

### 3. 主文件不要无限变长

原课建议 `SKILL.md` 保持清晰、简洁，必要时把额外内容拆到外部文件。

再具体一点，原课给出的经验值是：

- `SKILL.md` 尽量控制在 500 行以内
- 长格式模板、样例输出、领域材料尽量拆到外部文件
- 让主文件负责总控，让引用文件负责细节

这背后的思想很重要：

- 主文件负责“总控”
- 外部文件负责“细节”

这样 agent 更容易按需加载，维护起来也更轻松。

## 自定义 skill 的推荐写法

你可以按这个顺序思考：

1. 这个 skill 解决什么问题
2. 用户会在什么情况下需要它
3. 输入通常是什么
4. 输出应该长什么样
5. 流程步骤是否固定
6. 哪些边界情况必须提前说明
7. 哪些资源应该放到外部文件

如果你已经开始考虑跨环境复用，原课还有两个容易被忽略的提醒：

- 元数据里可以有可选字段，比如 `license`、`compatibility`
- 即使在 Windows 场景，也尽量统一使用正斜杠路径，减少跨环境兼容问题

## 一个最常见的误区

很多新手第一次写 skill，会把所有想法一次性都塞进去。

结果通常是：

- 主文件过长
- 步骤不稳定
- 触发描述模糊
- 后续难维护

所以原课一直在提醒你：先做最小可用，再慢慢补细节。

它甚至明确提到：一个系统里完全可以同时存在很多个 skill。重点不是做一个“巨无霸 skill”，而是把名字、边界和职责拆清楚。

## 常见误区

- 误区 1：以为 skill 写得越长越专业
- 误区 2：只写“做什么”，不写“什么时候用”
- 误区 3：把所有边界情况都留给模型临场发挥

## 原课真正想让你学会的，不只是格式本身

看起来这一章在讲字段和规范，但它真正训练的是一种“把工作流结构化”的能力。

你学的不是：

- 一个 frontmatter 该怎么写

你学的是：

- 怎样把隐性的经验，变成显性的、可复用的方法

这恰恰是很多中文用户最缺的一步。很多人已经会用 AI，但还没有把“自己怎么用顺手的”沉淀下来。

## 给零基础读者的一个务实建议

不要一开始就写“很大”的 skill。

先找一个你已经重复做过至少 3 次的小任务，比如：

- 把文章整理成提纲
- 审一段 SQL
- 给课程讲义生成复习题

先写出第一版，再慢慢加边界处理和外部资源。

## 本章你应该带走什么

- skill 的最小入口是 `SKILL.md`
- 好的 `name` 和 `description` 会直接影响 skill 是否能被正确使用
- 写 skill 的关键不是写多，而是写清楚流程、边界和触发条件

## 本章练习

- 选一个你最熟悉的小任务，先写 `name` 和 `description`
- 再列出 5 个以内的主步骤，看看它是否已经具备 skill 雏形

## 这章学完后可以立刻做什么

- 写出你自己的第一个 `name + description + 5 步流程`
- 把“什么时候用”这件事显式写进描述

## 这章最适合用在哪些场景

- 你准备开始真正动手写第一个 skill
- 你已经有一个重复任务，但总写不好描述
- 你想把自己脑子里的经验变成别人也能复用的方法

## 建议产出

- 一份 skill 初稿，至少包含：
  - `name`
  - `description`
  - 主体步骤

# 03 Skills、Tools、MCP、Subagents 怎么分工

## 这章讲什么

这一章是整套课里最容易混淆、也最值得弄懂的一章。

很多人会把这些概念混在一起：

- skill
- tool
- MCP
- subagent

如果你只记住一句话，那就是：

> 它们不是同一种东西，而是在 agent 系统里各自承担不同职责。

## 为什么这一章必须先分清

如果这一章没理顺，后面你很容易出现两种错误：

- 该补工具的时候去写 skill
- 该做分工的时候去硬塞更多提示词

所以这一章不是名词解释题，而是后面所有工程判断的底层分界线。

## 一句话区分

- `Tool`：给 agent 具体做事的能力
- `MCP`：把 agent 接到外部系统和数据上
- `Skill`：告诉 agent 应该怎样把这些能力组织成稳定流程
- `Subagent`：把任务拆给独立上下文去做

这里面第一次出现的两个词，建议先这样记：

- `MCP`：给 agent 接外部世界的接口标准
- `Subagent`：主 agent 叫来的“分工助手”

## skill 和 tool 的区别

原课用了一个很形象的比喻：

- tool 像锤子、锯子、钉子
- skill 像“怎么搭一个书架”

也就是说：

- tool 解决“你能做什么”
- skill 解决“你应该怎么做”

工具是底层能力，skill 是工作方法。

你可以把它再想成：

- tool 像零件
- skill 像装配说明

有零件不代表你就会装，有说明也不代表你手里已经有零件。二者缺一不可，但职责不同。

## skill 和 MCP 的区别

MCP 更像一根“连接线”。

当 agent 需要访问：

- 数据库
- Google Drive
- Notion
- 外部服务

这种模型本身不知道的数据和系统时，就需要 MCP 或类似接入方式。

所以你可以这样记：

- `MCP` 把数据拿进来
- `Skill` 告诉 agent 拿到数据之后要怎么处理

原课一句很重要的话是：

> MCP 提供 access，skills 提供 expertise。

这句话非常值得记住。

如果你觉得英文词太抽象，可以直接翻成：

- access = 接入能力
- expertise = 处理方法和专业经验

## skill 和 subagent 的区别

`Subagent` 是一种任务拆分机制。

当主 agent 发现一个任务太大，或者适合并行处理，它可以把子任务交给另一个独立上下文的 agent。这个子代理可以有自己的工具权限，也可以有自己的专长。

skill 和它的关系是：

- subagent 负责“谁来做”
- skill 负责“怎么做”

甚至一个 subagent 也可以在自己的上下文里使用 skill。

## 什么时候应该优先想 skill，而不是 subagent

如果你的问题是：

- 这类任务每次做法都差不多
- 输出标准应该统一
- 需要稳定方法而不是更多并行

那优先想 skill。

如果你的问题是：

- 一个任务太大
- 需要分角色
- 需要隔离上下文
- 需要并行处理多个方向

那更像 subagent 问题。

## 一个完整系统里，它们怎么一起工作

原课给出的理想组合大概是这样：

1. `MCP` 负责连外部数据
2. `Tool` 提供读写、搜索、执行等基础能力
3. `Skill` 规定方法和流程
4. `Subagent` 负责并行或分工执行

这套组合的价值在于：agent 不是只有“能做事”，而是开始具备“按方法做事”的能力。

## 一个更贴近现实的例子

假设你要做一个“每周客户反馈分析” agent：

- `MCP` 负责连上问卷系统、文档库或数据库
- `Tool` 负责搜索、读文件、执行脚本、导出报告
- `Skill` 负责规定分析步骤、分类标准和输出格式
- `Subagent` 负责分别处理访谈、问卷和工单数据

当你用这种方式去想，一个复杂系统就不会再变成一锅粥。

## 更贴近中文团队的场景例子

假设你在做一个“每周用户反馈周报”：

- 飞书文档、问卷平台、工单系统里的数据要先接进来，这更像 `MCP`
- 搜索、读取、分类、导出这些动作更像 `tool`
- “先去重、再分类、最后按固定模板汇总”更像 `skill`
- “访谈分析”“问卷分析”“工单分析”交给不同角色去做，更像 `subagent`

## 最实用的判断方法

如果你总分不清这几个概念，可以用下面这组问题自测：

- 我现在缺的是能力，还是缺方法？
- 我现在缺的是数据入口，还是缺处理规则？
- 我现在缺的是一个更会做的人，还是缺一个更清晰的流程？

对应关系通常是：

- 缺能力 → tool
- 缺数据入口 → MCP
- 缺流程方法 → skill
- 缺分工隔离 → subagent

## 常见误区

- 误区 1：把 MCP 当成一种“大号工具”
- 误区 2：把 skill 当成任何问题的万能答案
- 误区 3：觉得只要有 subagent，skill 就不重要了

## 本章你应该带走什么

- tool、MCP、skill、subagent 不是同义词
- 它们分别解决“能力、接入、方法、分工”四类问题
- 一个成熟的 agent 系统，往往是这四者协同工作

## 本章练习

- 找一个你熟悉的工作流，分别写出它需要的 `tool`、`MCP`、`skill`、`subagent`
- 如果你现在只能先做一个，判断它最优先缺的是哪一类

## 这章学完后可以立刻做什么

- 选一个真实工作流，给它做一次“四分法拆解”
- 明确哪里是能力问题，哪里是方法问题，哪里是分工问题

## 这章最适合用在哪些场景

- 你和同事讨论 agent 系统时经常概念混乱
- 你不知道一个问题该从工具、MCP 还是 skill 下手
- 你准备设计多组件协作的 agent 工作流

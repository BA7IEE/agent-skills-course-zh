# 09 结课总结与下一步

## 这章讲什么

这一章是整套课的收束。原课最后给出的建议很朴素，但很重要：

- 从简单 skill 开始
- 观察真实使用效果
- 再持续迭代

## 到这里，你应该已经完成了一个认知转变

在课程开始时，很多人会把 skill 看成：

- 一个新名词
- 一种更复杂的 prompt 写法
- 某个平台的新功能

学到这里之后，你更应该把它看成：

- 一种把方法稳定下来、把工作流沉淀下来的方式

## 回头再解释一个最核心的词

如果你学到这里还想再用一句话定义 `skill`，我会建议你这样说：

> skill 不是让 AI 变魔法，而是让 AI 在重复任务里更稳定、更像一个按方法做事的助手。

如果你想再顺手复习两个常见词，也可以这样记：

- `workflow`：一套有步骤、有顺序、可重复的工作流
- `agentic`：不是单次回答，而是会主动推进任务的那种工作方式

## 先复盘这套课真正讲了什么

学完整套课后，你应该已经形成这样一条主线：

1. 重复工作不该永远靠临时 prompt
2. skill 是把方法、规则和资源打包成稳定工作流的方式
3. skill 不等于 tool，不等于 MCP，也不等于 subagent
4. skill 可以跨不同环境复用，但不同环境的工程条件不一样
5. 当系统变复杂时，skill 更像“方法层”，而不是“小功能”

## 为什么原课最后强调“先简单，再迭代”

因为很多人学完后最容易犯的错误是：

- 一上来就想做一个很大的 skill
- 结果既难触发，又难维护
- 最后反而不如手写 prompt

更合理的做法是：

- 先写一个最小可用版本
- 在真实任务里观察它是否稳定
- 根据失败案例补边界规则
- 再慢慢拆出外部文件、模板和脚本

## 结课后最值得马上做的事

如果你想把这套课真正学会，而不是“看完就算了”，建议你立刻做下面三件事：

### 1. 选一个你自己最熟悉的重复任务

不要选太大，选你最近经常做的事，比如：

- 文章提纲整理
- PR 初审
- 数据表检查

### 2. 写出第一版 skill 草稿

至少写出：

- `name`
- `description`
- 主体步骤

### 3. 在真实任务里跑一次

不要只停留在“写完了”。真正有价值的是：

- 它有没有被正确触发
- 它是不是按你预想的方式执行
- 哪些边界情况还没考虑

如果你只把它停留在文档里，这套课就只学了一半。真正的理解，通常发生在你第一次把它拿去做真实任务的时候。

## 从中文工作场景出发，可以怎么开始

你完全可以从这些小事开始：

- 把“整理课程笔记”做成 skill
- 把“公众号文章改成提纲”做成 skill
- 把“每周数据复盘”做成 skill

这些事情不大，但非常适合训练你把方法固定下来的能力。

## 如果你继续往下学，推荐路线是什么

### 偏概念理解

继续补：

- MCP 标准与典型 server
- agent 任务拆分方法
- prompt 设计与 workflow 设计差别

### 偏开发实践

继续补：

- Claude Code / Codex CLI / 类似工具中的项目级工作流
- API 工具接入
- 多代理协作模式

### 偏产品落地

继续补：

- 如何把团队方法沉淀成 skill
- 如何做可维护的技能库
- 如何做版本管理和效果回收

## 本章你应该带走什么

- 真正重要的不是“会不会写 skill 文件”
- 而是你开始理解：怎样把重复工作变成稳定方法
- 这套思路不仅适用于 Claude，也适用于更广义的 agent 工作流设计

## 常见误区

- 误区 1：以为学完这套课就等于已经会搭完整 agent 系统
- 误区 2：以为 skill 只适合程序员
- 误区 3：只收藏课程，不把任何一个想法真正写成第一版 skill

## 本章练习

- 写出你准备做的第一个 skill 主题
- 用两句话说明它为什么值得被 skill 化
- 写出它最小可用版本需要包含的 3 到 5 步

## 这章学完后可以立刻做什么

- 从整套课里选一个你最想先实践的方向
- 今天就写出第一版 skill 雏形，不追求完整，只追求开始

## 这章最适合用在哪些场景

- 你准备结束“只学不做”的状态
- 你想把课程真正转成自己的产出
- 你要给自己定一个学完后的行动路线

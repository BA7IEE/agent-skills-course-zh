# 02 什么是 Agent Skills 与开放标准

## 这章讲什么

上一章你知道了“为什么值得做成 skill”。这一章要解决的是：

- skill 到底是什么
- 一个 skill 里通常会有什么
- 为什么原课反复强调“开放标准”
- 什么叫“渐进式披露”

## 这一章最容易卡住的点

很多人读到这里会开始被名词压住：

- 文件夹
- [`SKILL.md`](../resources/examples/skill-template/SKILL.md)
- 标准
- 渐进式披露

所以你读这一章时，不要一开始就追求把每个词背下来，而要先抓住主线：

> skill 是一种把方法、资源和执行规则组织起来的方式。

## 这里第一次出现的术语怎么理解

- [`SKILL.md`](../resources/examples/skill-template/SKILL.md)：skill 的主入口文件，你可以把它理解成“总说明书”
- `Markdown`：一种常见的纯文本写作格式，很多文档和说明文件都会用它
- `渐进式披露`：不是一开始把所有内容都塞给模型，而是先给最必要的部分，再按需要继续加载

## skill 到底是什么

最简单的理解方式是：

> skill 是一个文件夹，里面装着 agent 完成某类任务所需要的说明和资源。

这个文件夹里至少会有一个 [`SKILL.md`](../resources/examples/skill-template/SKILL.md)。

通常还可能包含：

- 其他 markdown 说明文档
- 可执行脚本
- 模板文件
- 图片、图标等素材

也就是说，skill 不是“单个文件”，更不是“某条 prompt 的保存版”，而是一个围绕任务组织起来的工作包。

## 为什么 skill 不只是文字说明

原课特别强调，skill 不仅能引用文字，还能引用脚本和资源文件。

比如 PDF 相关 skill，可能需要：

- 提取表单字段
- 把 PDF 转成图片
- 自动填写表单

这些事情靠说明文字不够，必须要有实际执行代码。所以 skill 可以告诉 agent：

- 什么时候该运行脚本
- 要运行哪个脚本
- 脚本负责做什么

这就是为什么 skill 更接近“工作流包”，而不是“说明书”。

## 为什么“开放标准”这件事值得单独讲

如果一个 skill 只能在一个产品里、一个按钮里、一个界面里工作，那它的复用价值其实很有限。

原课强调开放标准，是在说明：

- 你沉淀下来的不是某个平台专属小技巧
- 而是一套更通用的工作方法表达形式

对中文读者来说，可以把这理解成：

> 你写的 skill，不应该只服务某一个具体界面，而应该尽量写成“换个兼容环境也能继续用”的形式。

## 什么叫渐进式披露

这是 skills 里一个非常关键、也很容易被忽视的点。

如果一个 skill 很复杂，里面可能有很多：

- 主说明
- 参考材料
- 脚本
- 模板

如果每次对话一开始就把这些全部塞进上下文，模型成本会很高，信息也会很杂。

所以 skill 常见的工作方式是：

1. 平时只暴露“名字和描述”
2. 当用户任务和 skill 描述匹配时，再加载主体内容
3. 如果还需要额外资源，再进一步加载脚本和参考文件

这就叫渐进式披露。

## 中文场景例子

你可以把它想成公司新人入职：

- 第一天下发的是最基础的公司介绍和岗位说明
- 真要开始做某个专项工作时，才会继续给你 SOP、模板和历史案例

skill 的按需加载，和这种“先给必要信息、再给细节材料”的做法很像。

## 读到这里，最重要的认知转变是什么

很多人第一次接触 skill，会下意识把它理解成：

“哦，就是把一大段 prompt 存起来。”

但原课真正想强调的是：

- skill 有目录结构
- skill 有入口文件
- skill 可以引用外部资源
- skill 应该按需加载，而不是一口气全塞进去

换句话说，它更像一个轻量的“任务能力包”。

## 这一章和上一章的关系

上一章解决的是“为什么值得做成 skill”。

这一章解决的是“做成 skill 之后，它在结构上到底是什么”。

先有动机，再有对象定义，这样后面你才不会把 skill 和其他概念混在一起。

## 什么场景特别适合 skill

原课提到几类典型场景：

- 领域知识：品牌规范、法律审阅流程、数据分析方法
- 可重复流程：周报分析、客户准备流程、季度复盘
- 新能力：生成 PPT、生成 Excel 报表、构建 MCP server

你可以把它记成一句话：

> 只要一个任务需要“固定方法 + 固定资源 + 稳定输出”，它就很像 skill 的适用场景。

## 常见误区

- 误区 1：把 skill 理解成“存起来的一段超长 prompt”
- 误区 2：以为 skill 必须很大、很复杂才算 skill
- 误区 3：以为开放标准等于“所有平台效果完全一样”

## 本章你应该带走什么

- skill 是一个围绕任务组织起来的文件夹，不是一段孤立 prompt
- [`SKILL.md`](../resources/examples/skill-template/SKILL.md) 是入口，但 skill 不只包含它自己
- 开放标准意味着可迁移、可复用、可扩展
- 渐进式披露是 skill 能规模化使用的重要原因

## 本章练习

- 假设你要做一个“整理会议纪要”的 skill，写下这个 skill 目录里可能会有哪些文件
- 用你自己的话解释“渐进式披露”，不要照搬术语表

## 这章学完后可以立刻做什么

- 画出一个最简单的 skill 目录草图
- 判断哪些内容该放在 [`SKILL.md`](../resources/examples/skill-template/SKILL.md)，哪些该拆到外部文件

## 这章最适合用在哪些场景

- 你已经知道自己想做一个 skill，但不清楚它的结构
- 你总把 skill 和 prompt、模板、脚本混在一起
- 你准备以后让 skill 在不同环境里复用

## 建议产出

- 一个“我的第一个 skill 目录草图”，哪怕只有文件名列表也可以
